# 2026-03-30 시스템 정리

## 1. 전체 파이프라인

### CARLA → ROS2 (센서 데이터)

| CARLA 센서 | ROS2 Topic | 타입 |
|-----------|-----------|------|
| `sensor.lidar.ray_cast` | `/carla/hero/lidar` | `PointCloud2` |
| `sensor.other.imu` | `/carla/hero/imu` | `Imu` |
| `sensor.pseudo.speedometer` | `/carla/hero/speedometer` | `Float32` |
| `sensor.pseudo.odom` | `/carla/hero/odometry` | `Odometry` (ground truth) |
| `actor.pseudo.control` | 제어 명령 수신 | — |

### ROS2 내부 처리

```
/carla/hero/lidar (PointCloud2)
  → pointcloud_to_laserscan_node
    (min_height=-2.35, max_height=-1.8, range_max=15.0, angle_increment=0.0087)
  → /scan (LaserScan)

/carla/hero/speedometer + /carla/hero/imu
  → speed_imu_odom_node (odometry.py)
    - linear vel: speedometer
    - yaw: angular_velocity.z 적분 (use_imu_angular_velocity: True)
    - publish_tf: True (odom→hero TF 직접 발행)
  → /odom_wheel (Odometry)
```

### Nav2 파이프라인

```
waypoint_sender.py
  → NavigateToPose action
    → bt_navigator (navigate_to_pose_no_spin.xml)
        ├─ RateController (1Hz)
        │   → SmacPlannerHybrid (DUBIN) ← global_costmap (static map + /scan)
        │     → /plan (Path)
        ├─ RPP Controller ← local_costmap (/scan + odom)
        │     → /cmd_vel (Twist)
        └─ AMCL ← /scan + /odom_wheel (TF)
              → map→odom TF 보정 + /amcl_pose
```

### ROS2 → CARLA (제어 명령)

```
/cmd_vel (Twist)
  → cmd_vel_to_ackermann
    (wheelbase=3.0, max_steering_angle=0.6, min_speed_for_steer=0.1)
  → /carla/hero/ackermann_cmd (AckermannDrive)
    → carla_ackermann_control_node
      → /carla/hero/vehicle_control_cmd
        → CARLA vehicle
```

## 2. 사용 알고리즘

### AMCL (Adaptive Monte Carlo Localization)
- 역할: LiDAR scan + odom TF를 이용해 map 상 차량 위치 추정
- 입력: `/scan`, `odom→hero` TF (`/odom_wheel` 기반)
- 출력: `map→odom` TF, `/amcl_pose`
- 핵심 파라미터:

  | 파라미터 | 값 | 설명 |
  |---------|-----|------|
  | `robot_model_type` | `DifferentialMotionModel` | Ackermann이지만 2D 근사 |
  | `min_particles` | `500` | 최소 파티클 수 |
  | `max_particles` | `2000` | 최대 파티클 수 |
  | `laser_model_type` | `likelihood_field` | scan 매칭 모델 |
  | `update_min_d` | `0.1` | 이동 시 파티클 업데이트 최소 거리 |
  | `update_min_a` | `0.1` | 회전 시 파티클 업데이트 최소 각도 |

- 이슈: odom drift가 클수록 AMCL 보정 부담 증가. IMU noise stddev=0.05 이상에서 localization 오차 급증 확인

### SmacPlannerHybrid (Global Planner)
- 역할: goal까지 kinematic-feasible 경로 생성 (Hybrid A*)
- 핵심 파라미터:

  | 파라미터 | 값 | 설명 |
  |---------|-----|------|
  | `motion_model_for_search` | `DUBIN` | 전진만 허용 (REEDS_SHEPP는 후진 너무 공격적) |
  | `minimum_turning_radius` | `3.0 m` | 플래너 최소 회전 반경 |
  | `cost_penalty` | `5.0` | 장애물 근처 경로 비용 |
  | `downsample_costmap` | `true` (factor 2) | 계획 속도 향상 |
  | `tolerance` | `0.5 m` | goal 도달 불가 시 허용 대안 거리 |
  | `max_planning_time` | `10.0 s` | 최대 계획 시간 |

- 이슈: goal orientation과 접근 방향이 다를 때 우회 경로 생성. `yaw_goal_tolerance`는 goal checker 설정이라 planner에 영향 없음 → waypoint orientation을 자연스러운 접근 방향으로 설정해야 함

### RPP Controller (Local Controller)
- 역할: 경로 추종 및 속도 명령 생성
- 핵심 파라미터:

  | 파라미터 | 값 | 설명 |
  |---------|-----|------|
  | `desired_linear_vel` | `0.5 m/s` | 목표 속도 |
  | `min_approach_linear_velocity` | `0.15 m/s` | goal 근처 최소 속도 |
  | `lookahead_dist` | `2.0 m` | 경로 추종 lookahead 거리 |
  | `use_regulated_linear_velocity_scaling` | `true` | 곡률에 따라 속도 자동 감소 |
  | `allow_reversing` | `false` | 후진 비활성화 |
  | `use_rotate_to_heading` | `false` | Ackermann 차량 제자리 회전 불가 |

- 이슈: 좁은 구간에서 속도 scaling 과도 작동 → `Failed to make progress` 발생 가능

### Goal Checker
- `xy_goal_tolerance: 0.5 m`
- `yaw_goal_tolerance: 2.0 rad` — 방향 무관, 위치만으로 도달 판단
- `stateful: True`

### Progress Checker
- `required_movement_radius: 0.2 m`
- `movement_time_allowance: 15.0 s`
- 15초 동안 0.2m 못 움직이면 FollowPath 실패 → recovery 발동

### Behavior Tree (navigate_to_pose_no_spin.xml)

```
RecoveryNode (retries=8)
  ├─ PipelineSequence
  │   ├─ RateController (1Hz)
  │   │   └─ RecoveryNode (retry=1)
  │   │       ├─ ComputePathToPose
  │   │       └─ ClearGlobalCostmap (fallback)
  │   └─ RecoveryNode (retry=1)
  │       ├─ FollowPath
  │       └─ ClearLocalCostmap (fallback)
  └─ ReactiveFallback
      ├─ GoalUpdated
      └─ RoundRobin
          ├─ BackUp (dist=1.0m, speed=0.2m/s)   ← 홀수 retry
          └─ Wait (1초)                           ← 짝수 retry
```

- retries=8 → BackUp 최대 4회
- Spin 제거: Ackermann 차량 제자리 회전 불가
- ClearCostmap recovery 제거: 좁은 공간 실제 막힘이 원인이라 효과 없음
- BackUp 방향 전환 지연 이슈: Ackermann 차량 전진→정지→후진 전환 시 속도 0 필요

## 3. 파라미터 튜닝 히스토리

| 파라미터 | 변경 전 | 변경 후 | 이유 |
|---------|--------|--------|------|
| `motion_model_for_search` | `DUBIN` | `REEDS_SHEPP` → `DUBIN` 복귀 | REEDS_SHEPP 후진이 너무 공격적 |
| `minimum_turning_radius` | `5.5` | `3.0` | 5.5m는 대형 트럭 수준, 좁은 공간 회전 불가 |
| `cost_penalty` | `8.0` | `5.0` | 장애물 근처 우회 경로 과도 생성 감소 |
| `yaw_goal_tolerance` | `1.0` | `2.0` | goal orientation mismatch로 인한 재계획 감소 |
| `bt_navigator odom_topic` | `/odom_local` | `/odom_wheel` | wheel odom 기반 전체 Nav2 동작 |
| `velocity_smoother odom_topic` | `/odom_local` | `/odom_wheel` | 동일 |
| `movement_time_allowance` | `15.0` | `5.0` → `15.0` 복귀 | 5초는 너무 짧아 정상 주행 중 false positive |
| `required_movement_radius` | `0.2` | `0.1` → `0.2` 복귀 | 기본값이 적절 |
| `BackUp dist/speed` | `0.3m / 0.05m/s` | `1.0m / 0.20m/s` | 0.05m/s는 방향 전환 timeout 원인 |
| `number_of_retries` | `6` | `8` | BackUp 4회 실행 가능하도록 |
| `Wait wait_duration` | `2s` | `1s` | 불필요한 대기 시간 감소 |
| `ClearCostmap (recovery)` | 활성화 | 주석처리 | 좁은 공간에서 효과 없음 |
| `odometry.py yaw 방식` | `msg.orientation` (절대 yaw) | `angular_velocity.z` 적분 | 실차 전환 준비 + IMU 노이즈 반영 |

## 4. Waypoint 정보

### 경로: WP1 → WP2 → WP3 → WP4 (순환)

```python
WAYPOINTS = [
    (46.626,  0.000, -0.131, 0.991),   # WP1: yaw -15°
    (48.180, -16.008, -0.892, 0.452),  # WP2: yaw -126°
    (11.741, -17.762,  0.991, 0.131),  # WP3: yaw +154°
    (20.609,  -0.391, -0.020, 1.000),  # WP4: yaw -2°
]
```

| WP | x | y | yaw | 위치 |
|----|---|---|-----|------|
| WP1 | 46.626 | 0.000 | -15° | 주차장 우측 상단 |
| WP2 | 48.180 | -16.008 | -126° | 주차장 우측 하단 |
| WP3 | 11.741 | -17.762 | +154° | 주차장 좌측 하단 |
| WP4 | 20.609 | -0.391 | -2° | 주차장 중앙 상단 |

### 주의 구간
- **WP2 goal 근처**: orientation(-126°)이 접근 방향과 달라 플래너가 우회 경로 생성 가능
- **WP3 goal 근처**: yaw(154°)가 많이 꺾여 있어 동일 문제 가능
- **좁은 통로 구간**: 주차된 차량 사이 통과 시 `Failed to make progress` 발생 가능
- **BackUp 발동 시**: 전진→후진 방향 전환 지연으로 backup 실패 가능 (속도 완전 감소 필요)
